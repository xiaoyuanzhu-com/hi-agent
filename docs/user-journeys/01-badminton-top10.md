# 羽毛球男单世界前十

**Persona:** 普通用户,第一次随口和 agent 聊天,没有任何使用负担,默认用语音 + 看屏幕。
**Goal:** 打个招呼,然后让 agent 给自己讲讲一个需要联网查询的话题(前十球员),并在过程中自由追问、切换话题。
**Preconditions:** agent 在线、能听能说;具备联网搜索能力;屏幕作为 agent 的演示舞台(见 [overlay 呈现指南](../overlay-presentation-guideline.md))。

---

## Steps & expected UX

整个场景是**一段连续的对话**,屏幕状态层层递进:闲聊 → 演示前十 → 钻取单个球员 → 切换到天气。下面按"幕"用表格组织。

> 文字回复即语音回复(同一份内容,合为"回复");UI 与转场合为一列。

| 幕 | 用户输入 | 回复(文字 = 语音) | UI 与转场 | Principle |
|---|---|---|---|---|
| **0 · 打招呼** | "你好" | 简短一句(如"嗨"),极快,像随口接话 | 安静/待命态;无转场 | 对话简短、拟人;不寒暄、不问"有什么可以帮你" |
| **1 · 提出需时查询** | "我想看羽毛球男单世界前十的球员" | "好,我去查一下" | 待命/轻 loading 态;可有轻微 loading 指示 | 思考与接话解耦:先接住话,检索/思考异步在后台进行([[channels-as-autonomous-agent]]) |
| **2 · 检索完成 + 演示前十** | —(系统:检索返回) | "查好了" 起头,随即一边讲一边演,音画结合像讲 PPT;**递减压缩**、不啰嗦(见备注 A) | 前十呈现,每张卡片**信息丰富**:海报/形象 · 主要成就 · 排名;**窗口式轮播**——窗口滑动 `[1,2,3]→[2,3,4]→[3,4,5]`,主位放大居中、旧的缩回场边,带前后缓冲(见备注 B) | 演示 = 叙述 + UI;递减压缩;窗口式轮播 + 松对齐;信息全在屏上 |
| **3 · 收尾** | —(演示讲完) | 无,答完即停 | 维持前十终态;无转场 | 克制收尾:不说"接下来想干嘛""还想看别的吗" |
| **4 · 钻取单个球员** | "我想再看一下石宇奇" | 着重介绍其主要成就 + 当前状态,简短 | 石宇奇放大为主体、其余前十退为背景;**钻取转场**:前十这组 UI 缩小,石宇奇放大出来 | 柔和转场:总览 → 单项用"缩小总览 / 放大单项" |
| **5 · 换域(换 View)** | "今天天气怎么样?" | 配合天气内容的简短介绍 | 天气 View 取代羽毛球;**换域转场**:羽毛球 UI fade out / 引退,天气 View fade in | 柔和转场:换域用 fade out → fade in,始终配合语音 |

**备注 A · 递减压缩的叙述节奏**:第一名(石宇奇)讲得相对完整("第一是石宇奇 + 他的主要成就");第二、第三逐渐变短、变快;**后面几个可以越来越简短,甚至不逐一口播**——UI 上展示完即可。

**备注 B · 窗口式轮播(carousel),不是精确逐人对齐**:这是有意为之——语音和"当前讲到第几个"在技术上**很难精确对齐**,暂不强求。布局上有一个**"当前主位" + 若干"场边位"**;推进时按窗口滑动(`[1,2,3]→[2,3,4]→[3,4,5]`…);即使口播已经讲到第二个、UI 才刚切,也**没关系**——窗口自带**前后缓冲**,容得下这种漂移。**所有信息都在屏幕上**,用户不必动手翻找,只在想回看时才打断。

## Expected outcome

- 用户全程**只靠说话和看屏幕**就拿到了完整信息,没有被要求动手翻页/点击。
- agent 的口播**简短、像真人**:快速接话、异步检索、答完即止,没有客套和填充语。
- 演示是**音画结合的连续演出**,过渡有动画、有缓冲,观感顺滑,即便语音/UI 不精确对齐也不出戏。
- 话题推进(总览 → 钻取 → 换域)都有**明确而柔和的转场**,屏幕状态始终反映"当前在讲什么"。

## UX principles this journey establishes

这些是可被后续 journey 复用的通用原则(不止于羽毛球):

- **对话简短、拟人**:快速接话,答完即停,不说"接下来想干嘛"这类填充语。
- **思考与接话解耦**:需要时间的任务,先给一句简短确认接住话,检索/思考在后台异步进行,完成后再简短告知并开演。
- **演示 = 叙述 + UI**:屏幕是 agent 的演示舞台,音画结合,像讲 PPT。
- **递减压缩的叙述**:重点项讲得完整,后续越来越短、越来越快,可不逐项口播,剩余信息交给 UI。
- **窗口式轮播 + 松对齐**:用"主位 + 场边位"的滑动窗口承载语音/UI 的不精确对齐,窗口自带前后缓冲,容忍漂移。
- **信息全在屏上**:不需要用户动手翻找。
- **柔和转场**:钻取用"缩小总览 / 放大单项",换域用 fade out → fade in,始终配合语音。

## Edge cases & failure modes

- **检索很慢/失败** → 仍需先有"我去查一下"的接话;失败时简短如实告知,不堆砌错误细节(用户侧不弹错误卡片,见 [[diagnostics-belong-server-side]])。_(待补:具体话术与降级 UI)_
- **语音/UI 漂移过大** → 窗口缓冲是兜底;若漂移超出窗口,倾向于让 UI 等一等而非追帧。_(待定阈值)_
- **用户在演示中途打断**(追问/切话题)→ 应能平滑接住,当前演示让位于新意图(第 4、5 幕即是例子)。

## Open questions

- 语音和"当前主位"对齐到什么粒度算"够用"?窗口大小(3 个?)和缓冲策略要不要可调?
- 钻取/换域的动画时长与节奏,是否有统一的转场规范?
- "递减压缩"到第几名开始不再口播,是固定的还是按内容/时长自适应?
- 多客户端场景下,这套演出在每个客户端都应渲染出**一致**的画面(见 [[in-out-channel-namespace]])——轮播窗口的状态如何同步?

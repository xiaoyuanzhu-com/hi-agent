# 去体验个产品的新功能,剪条小红书发出来(测试即素材)

**Persona:** 老板随口把一件"体验 + 传播"的活整个交出去 —— 让 agent 自己去某个产品(这里:pi.dev)上真机体验新功能,边测边录,剪成一条小红书笔记发出去。**测试跑通的那段过程,本身就是营销素材。**
**Goal:** agent 在一台**能碰到的设备**(这里:Mac mini)上真操作 pi.dev、录下来、判定跑没跑通、干净的那次剪成成片(视频 + 封面 + 文案)、**发布前停下过目**、拿到"go"后操作**登录态的**小红书把它发出去。
**Preconditions:**
- 有设备句柄:Mac mini 在场(`ssh macmini`)且**有 GUI/受控桌面 + 录屏/辅助功能授权**(SSH 无头会翻车,见 [08](08-operate-apps.md) 实测);或经设备接口(abacad 式 MCP:`screenshot+tree → tap/type`)拿到句柄。
- 目标账号**已在该设备登录**:小红书(浏览器创作平台 或 手机 app)。**登录态 = 凭证,不走 API、不存密钥。**
- 意图来自用户:授权测这个产品、授权以这个账号发布。

这条把前面几路缝起来:操作设备([08](08-operate-apps.md))、浏览器办事([07](07-browser-errand.md))、看懂一段过程([27](27-watch-and-guide.md))、先研究再动手 / 收工先自评([22](22-research-before-stale-answer.md)/[23](23-critique-before-shipping.md))、看火什么的调性([04](04-trending-feeds.md))。新的只有一件事:**把一次真操作录下来、剪成对外的成片、并操作 app 发出去。**

## Steps & expected UX

1. **"pi.dev 新出的那个功能你去体验下,剪条小红书发出来"** → agent 先**对齐最小几点**(别多问,见 bias-to-action):测哪几个点、笔记调性(测评 / 上手 / 种草)、**发布前要不要过目**(默认要)。
2. **规划一条能看的故事线** —— 一个**连贯 journey**,不是测试矩阵。一条清楚的叙事("拿它做成 X")比覆盖率更重要:成片要好看,不是跑清单。
3. **开录** —— 在设备上起屏幕录制:Mac 用 `ffmpeg -f avfoundation` / `screencapture -v` / 系统自带录屏;手机则**操作它自己的系统录屏**(下拉 → 点录屏 → 自己点掉"开始录制"授权弹窗)。录制**落在设备本地**,不是实时流回来。
4. **边做边讲** —— 用 vision 看屏 → 点 → 看结果(see→act 回路,[08](08-operate-apps.md));同时**口播**("现在打开 pi.dev 的 X,点这里……")。口播即旁白素材。每一步动作带**时间线**(点了什么、几点几分),供后面对齐字幕/剪辑。
5. **判定(测试与营销不合流)** —— 干净跑通 → **可发**;撞 bug / 回归 → 落成 `drive/` 里一份 bug 报告(或记忆 jot),**不发**。红的那次不变成笔记,只变成反馈。这道岔口是"自动测"和"自动发"不塌成"跑出啥发啥"的关键。
6. **剪(ffmpeg 迭代,决策是 agentic)** ——
   - **裁**:按动作时间线砍掉等待 / 加载 / 重试 / 手忙脚乱(哪几段有用 agent 自己判)。
   - **字幕/高亮**:按时间线打步骤字幕、给点击处加圈/箭头。
   - **旁白**:逐行 TTS **渲成音频文件** → 量时长(ffprobe)→ 垫/裁画面让台词落在对的帧 → 混到画面下。**是个迭代对齐的 loop,不是一次算准。**
   - **导出**:小红书竖版(9:16)+ 封面图 + 若干配图;标题卡 / 片头片尾按需。
7. **发布前停** —— 把成片 + 封面 + 文案**摆成 view 预览**,等一句"go"。发布是对外、且实际不可逆(会被缓存/索引,删了也留痕),这一步**一定过人**。
8. **Go → 操作登录态的小红书真发** —— 不是调 API,是**像人一样操作那个已登录的 app**:浏览器创作平台上传视频+封面+文案,或手机 app 走 vision 一步步点。发完回执笔记链接。

## Expected outcome

- 一条**真发出去**的小红书笔记(视频 + 封面 + 文案),内容来自一次**真跑通**的 pi.dev 新功能体验;
- **或者**一句诚实的"这次没跑通,给你在 drive 落了个 bug 报告,没发"——绝不把没跑通的糊成笔记发出去。
- 发布这一步永远是"摆出来 → 等 go → 才发",没有静默对外。

## Edge cases & failure modes

- **Mac mini SSH 无 GUI/TCC** → 录屏 / 操作可能无权限(见 [08](08-operate-apps.md) osascript 翻车实测):**如实降级,别假装录了**。需用户在物理桌面一次性授予"屏幕录制 / 辅助功能"。
- **pi.dev 改版 / 控件变动** → 重试一次;不稳则如实降级,不硬撑。
- **手机端盲区**:`FLAG_SECURE`(银行类)/ canvas / 游戏帧空白 → 靠 tree,或改用桌面录。系统录屏是 Android 11+,录屏 tile 各家 OEM 位置不一(agent 靠 vision 找)。
- **小红书 web 创作平台限制**(时长 / 格式 / 审核)→ 按平台规格导出;审核不通过如实回报,不谎报已发。
- **TTS 渲不成文件** → 退化为**纯字幕版**(无旁白),别卡住。
- **发布不可逆** → 无"go"不发;拿不准宁可不发。
- **授权边界** → 只测 / 只发用户**明确授权**的产品与账号;换产品、换账号要重新确认。

## Open questions

- journey 由 agent 自己挑,还是每次用户点名测哪个功能?倾向:**agent 挑,用户只把发布关**。
- 没跑通的那次,要不要也能对外发("我们抓到个 bug / 这个功能还不行")—— 还是严格只进内部?决定发布岔口是一条路还是两条。
- Mac mini 录屏 / 辅助功能的一次性授权,怎么引导最顺(TCC 弹窗在物理机才能点)。
- 手机小红书 app vs 浏览器创作平台,哪个作默认发布面?
- 封面 / 文案的调性,靠 [04](04-trending-feeds.md) 现查当下小红书在火的版式来定,还是给固定模板?

## 本次能力盘点(2026-07-20 讨论,尚未建/未测)

把这条拆到底,**核心回路没有硬缺口**——都是 agent 驱动**已有的** effector:

| 环节 | 结论 |
|---|---|
| 碰到"不是自己这台"的设备、看屏 + 动手 | **abacad 式设备接口**(MCP:`screenshot+tree → tap/click/type` + `run_command` SSH + `/blobs` 传文件)闭合;非 API,是操作 UI。 |
| 录成一段视频 | **桌面**:`run_command` 起真录屏器 → `/blobs` 拉整文件,近乎白得。**手机**:操作它**自带的系统录屏**(agent 自己点掉授权弹窗),录完拉文件再剪——不需要给设备接口加 MediaProjection。 |
| 剪辑(裁 / 字幕 / 旁白 / 导出) | **不是能力缺口**:agent 会看懂素材 + 会驱动 ffmpeg。判定与剪辑决策是 agentic(该沉成一条 skill 求稳,而非每次现编 filtergraph)。 |
| 旁白对齐 | **迭代 loop**:TTS 渲文件 → ffprobe 量时长 → 垫画面 → 复核。唯一要**确认存在**的小依赖:**TTS 能渲成文件**(非只live朗读)。 |
| 发到社交平台 | **不是 API 缺口**:操作**登录态的 app**(浏览器创作平台 / 手机 app),登录态即凭证。等于回到"操作设备"这一路,已闭合。 |

_effector = 设备接口(abacad 式 MCP / Mac vision+ssh)+ 设备自带录屏或 ffmpeg + TTS-to-file + ffmpeg 剪辑;判定/剪辑决策 = agentic(建议沉 skill);发布 = 操作登录态 app(**非 API**)。**唯一待确认的原语:TTS 渲文件**;待打通的授权:Mac mini 录屏/辅助功能 TCC。成熟度:未建、未测。_

# 我要传你点东西,怎么弄(直接摆出上传入口:拖拽区 + 二维码)

**Persona:** 用户手里有要给 agent 的东西——一份合同、一张护照照片、一个文件——不知道怎么递过去。东西可能在当前这台机器上,也可能在手机里。这是最常用、最基础的一步:把东西交给它。
**Goal:** agent 立刻给出"怎么传"的入口,而且**优先直接在屏上摆出两个 view**——一块拖拽区、一个二维码,用户挑顺手的用,而不是念一段"你可以……"的说明。
**Preconditions:** agent 有自己的页面(屏在场)、能 `show_view`;有上传 carrier(web upload 端点 + 二维码→手机上传页,均 net-new)。文件是**递来的物件、按引用传**,不走 vision 感官(见 [19](19-upload-passport.md))。底层模型见 [data-dir-layout](../data-dir-layout.md)。

## Steps & expected UX

1. **"我需要传你一点东西,怎么弄"** → 接住,听懂这是"要把文件递给我"——不是要我描述、不是要我看,是要交付一个物件。
2. **优先摆出入口,而不是口述** → show 两个 view + 一句话点一下:
   - **拖拽区**:"把文件拖到这儿"——当前这台机器上的文件,直接拖进来。
   - **二维码**:手机扫开一个上传页——东西在手机里(相册的护照照、聊天里存的合同)就扫码发。
   - 一句:"拖这儿,或扫码用手机传都行。"
3. **入口绑当前 scene/会话** → 传上来的东西落到正确的对话里,agent 当场知道"刚收到你递来的东西"(接 [19](19-upload-passport.md))。
4. **用户挑一个用** → 之后怎么接、怎么存、怎么回话,见 [19](19-upload-passport.md)。

为什么优先"摆 view"而不是"念选项":屏是 agent 的演示([[overlay-presentation-model]])。"怎么传"这种带可操作入口的回答,摆出来即用 > 口述"你可以拖拽或扫码"。给一句引导 + 两个能立刻点的入口。

## Expected outcome

- 用户**当场看到能用的入口**(拖拽区 + 二维码),不必追问"具体怎么弄"。
- 手机里的东西也传得了(扫码桥接设备),不被"东西不在这台机器上"卡住。
- 入口绑对话:传上来的文件落到当前 scene,agent 接得住(续 [19](19-upload-passport.md))。

## Edge cases & failure modes

- **东西在手机、agent 在桌面** → 二维码就是为这个(桥接设备);别只给拖拽区。
- **其实只想发一句话 / 一个链接** → 直接说就行,不必走文件入口(文本本就能直接发)。
- **一次多个文件** → 拖拽区收多个;一句话确认收到几个。
- **纯文本通道、摆不了 view**(无屏 scene)→ 降级:给一个上传链接,或走已连接的 app(飞书那边本就能发附件,见 [02](02-feishu-sprint-backlog.md)/[03](03-feishu-flash-cards.md));如实说"这边给你个链接传"。
- **通过已连接 app(飞书)来的对话** → 那边能直接发附件,不必摆二维码,"发我就行"即可。

## Open questions

- 二维码上传页的鉴权与时效:绑 scene 的一次性 token、多久过期?——别让谁扫了都能往这个对话塞东西。
- 拖拽区是常驻一个小角标入口,还是只在"要传东西"时按需摆?(倾向按需,常驻可能碍事。)
- 要不要支持传给**非当前设备**的 scene(我在公司,想传到家里那台的 agent)?
- carrier 先做 web upload 还是直接接飞书?顺序未定(见 [data-dir-layout](../data-dir-layout.md) / [[file-exchange-drive-carriers]])。
- "预置内置" vs "按需生成"的边界划哪?上传归内置(基础常用 + 需后端端点);更花哨/小众的流程留按需。

_机制:carrier(web upload 端点:拖拽 + 二维码→手机上传页,绑 scene)+ `show_view`(把入口摆上屏)。文件 = 递来的物件、按引用传,不走 vision(见 [19](19-upload-passport.md))。可行性:可行。成熟度:**carrier = 上传 view + 后端端点,定为**预置打包成内置基础功能**(作为 seed,不是按需生成)——这是基础常用件,且新后端路由本就不能在运行时热加(只有 view/技能/知识能按需),所以端点必然内置,view 也随之预置一个统一默认,但仍是 `views/` 里的普通 view、agent 可再扩展。"摆入口"的 `show_view` 行为已有。**_

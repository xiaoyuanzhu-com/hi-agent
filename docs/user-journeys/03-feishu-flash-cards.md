# 飞书群 flash-cards → 记忆卡片图

**Persona:** 老板(用户),日常用飞书;有一个叫 flash-cards 的群,自己平时往里丢英语单词。希望 agent 把每个词做成一张**记忆卡片图**回到那条消息下面。和 agent 的关系是老板和员工:老板出任务、给必要的输入,员工干全部实际的活。
**Goal:** 常驻委托:盯住 flash-cards 群,每条消息(基本是单词)生成一张记忆卡片(图片),回复在原消息下。
**Preconditions:** hi-agent 在跑,能执行命令行工具、能联网。**飞书侧零现成接入**——没装 CLI、没有凭证;把工具置备出来是任务的一部分。老板能在浏览器里完成飞书扫码/授权。

> 本 journey 基于 2026-06-10/11 的真实运行提炼成**预期 UX**:主流程写理想行为——老板只给口味反馈,从不需要替 agent 抓质量;当次实际跑出的差距记在文末"实测缺口/复测"。与 [02](02-feishu-sprint-backlog.md) 的关键差别:这里的产出是**创作性的**(卡片有版式、有审美),且每张都直接发到群里——所以多出"样稿校准"一整段。

## Expected outcome

- 群里每条单词消息下面有且只有一张卡片,风格统一(老板选定那版)。
- 群成员看到的第一张卡片就是老板认可的成品——**校准全程在群外**。
- **老板的反馈只关乎口味**(版式、配色、信息密度),从不需要替 agent 抓质量问题。
- 老板的全部成本:一段委托对话、三次浏览器授权、一次选样稿;之后再没说过第二遍。
- 重启、断线之后不重复、不遗漏,**自己恢复,老板无感**。

## UX principles this journey establishes

- **交付必检是内建动作,不是被纠正后的补救**:交出去的东西要么是跑熟的路,要么发前亲自打开看过;生成的图片要亲眼看渲染结果。万一真出了错:认错不找借口、当场改——但那是兜底礼仪,不该出现在常规流程里。
- **样稿先行,风格校准**:风格/审美类公开产出,先拿真实样本做几版给老板选,确认前对外零动作。
- **要老板动手的内容必须在老板的通道里**:命令、链接、清单,完整打在文字里;"照着屏幕做"之前先确认对方有屏幕。
- **长活不占线,完工要交差**:批量工作派 worker,brain 随时接得住话;干完主动报完成与异常——沉默是对群消息的,不是对老板交办的。
- **自己的摊子自己管**:自己布置的进程坏了直接修,不请示;找老板只为只有他能做的事(授权、扫码)。
- **台账落盘**:处理过什么持久记录;恢复 = self 笔记 + 台账 + journal,不是老板重新交代。
- **先看真实数据再设计**;**员工用事实顶回老板**。

## 实测缺口(2026-06-10/11 运行 vs 上述预期)

当次运行委托、置备、校准段基本达标;以下是没达标的,即 hi-agent 待改进项:

1. **补齐 19 词在 reactor turn 里干了 17 分钟**,期间不可达,且完工没交差、smuggled 失败被吞 → 派 worker + 交差应进 core 引导。
2. **监听进程做完补齐后始终没拉起**,无人发现;重启后也无自恢复,靠老板上门 → pulse + hot.md 承诺机制(设计已有,未建)。
3. **三次把"要老板动手的内容"丢在老板看不到的地方**(指南只在 view 上、命令/URL 说到冒号就断)→ 通道在场感 + core 引导。
4. **第一版样稿带渲染错误直接发出** → 交付必检进 SOUL。
5. **被问"要不要我帮你启动"** → 所有权引导。
6. Host bugs:scene loop 卡死(swap 无超时阻塞循环)、utterance 跨整个 turn 不关闭(造成 long-poll 超时 + 重放)。
7. **agent 自写的 prompt override 无处生效**(core.local.md 躺在 workspace;实际上 data/prompts/core.local.md 槽位已存在,agent 没装进去)→ 自我引导落到 self.md。

### 复测(2026-06-11,serving-fixes 之后)

修复并验证通过:

- **pulse 自检是真的**:读 self 笔记 → `pgrep` 真查监听 → 正常则一言不发(转录可证)。
- **断后自愈全程无人参与**:杀掉监听 + 重启主机 → warm-up/首个 pulse 读到 self.md 里的职责("Must restart on every host process restart")→ 13 秒内自己把监听拉起来。
- **台账诚实**:被问 smuggled 时真去查台账,承认发送失败,当场补发并回 thread。
- scene 卡死(swap 超时 + 弃用宕死 session)、utterance 长开(3s quiet-close)、presence 进 prompt,均已修复生效。

尚待复跑验证:**交付必检**(core.md 已加 "ship only what you've seen",但样稿校准段还没在新引导下重演——预期下次 journey 测试首版即干净)。

### 清场复跑(2026-06-11,进行中)

全新环境(删库、删 lark-cli 与凭证、零记忆)从 act 1 重走。已验证:act 1-2 ✅(接住、调研、装 CLI 前请示);**act 3 的 URL 落文字 ✅**(两次主动 inline,零纠正——上轮最频繁的翻车点已不再出现)。中断于首次扫码(老板不在场),acts 4-11 待续。备注:~/.agents/skills 里的 lark 技能为上轮遗留未清,不影响关键复测项,但本轮不算 100% 纯净。

复测新发现(待改进):

- **职责需要单一 owner 或幂等启动**:两个 scene 的 warm-up 同时发现监听已死、同时拉起 → 双进程、消息双发(输出靠判断去重兜住,但进程泄漏无人察觉——pulse 的 `pgrep` 看到"有在跑"就满足,没数有几个)。
- **"空检查结果当健康"出现两次**:`curl`/`ps` 返回空却叙述"一切健康"——证据要读内容,不是读"命令跑完了"。已加 core.md 引导("a liveness probe that returns nothing means the thing is DOWN"),待观察。
- **职责必须落 self.md 才有恢复**:接受常驻委托的当下就该写;这次是老板提醒一次后才记(记完即生效)。
- 回复语言偶尔与老板语言不一致(中文问、英文答)——register 细节,SOUL 层打磨。

## Edge cases & failure modes

- **同一个词发两遍** → 台账查得到,不重复生成;回链接或简短说"做过了"。
- **生成失败** → 不发废稿;重试,连续失败记台账并在交差/pulse/过问时如实说。
- **一次灌入大量词** → 限速,别刷屏;拿不准先问。
- **鉴权过期** → pulse 发现,主动找老板:"需要你重新扫一次",说人话、说一次。

# 打断:我一开口,它就让路

**Persona:** 普通用户,正用语音和 agent 聊天 / 看它演示;说到一半想插话——追问、纠正,或干脆换个话题。
**Goal:** 像跟真人说话一样自然地插话:不必等它说完,不必找"停止"按钮;我插进去,它就让路,然后顺着我的话走。
**Preconditions:** agent 在线、能听能说;语音输出开着(纯文字场景见边角)。**这是所有语音对话的底座**——01 的演出、06 的主动浮现都跑在它上面。

---

## Steps & expected UX

整段交互的核心是一句话:**脑子可以一边说一边想,但嘴只有一张。**

1. **它正说着**(可能还在想后面几句)→ 边说边在屏上滚字。**字跟着嘴走,不抢跑**:不是把整段回复一次性糊上屏、让声音在后面追,而是一句一句地出、大致贴着语速(略微领先没关系)。屏幕反映的是"现在说到哪",不是"它打算说什么"。

2. **我开口插话** → 它的声音**当场停下,停在半句也没关系**——那是它在听我说话的样子,不是出故障。与此同时,**还没出口的那些话不再往外冒**:既不继续念,也不继续滚字。不会出现"我都在说别的了,它那边还一句接一句往上堆"的拥挤感。

3. **它接住我的话** → 它知道**自己刚说到哪、哪些没说出口**,也知道**我刚说了什么**,据此重新组织:
   - 我说的**跟它正讲的有关**(追问、纠正)→ 先答我,必要时把更新后的内容接着说。
   - 我**换了话题** → 当前要讲的让位给新意图;没说出口的尾巴**值不值得再说,它自己判断**——多数时候直接放掉。不复读、不来一句"刚才说到哪了",除非真有必要。

## Expected outcome

- 插话**像跟真人说话**:我一开口它就停,屏幕和声音一起静下来,没有抢话、拥挤、复读的机械感。
- 我**永远不需要等它说完**,也不需要任何"停止/取消"动作。
- 话题**平滑让位**:它顺着我的新意图走,而不是固执地把原话讲完。

## UX principles this journey establishes

- **嘴是单例,脑子是异步**:mind 可以继续思考、继续产出后面的话,但说话是串行的——多句话排队一张嘴说,绝不重叠。
- **字跟着嘴走**:屏上的口播文字贴着语速逐句揭示,不抢在声音前面糊一屏(估算级即可,略领先优于滞后)。这取代了"文字必然抢跑、靠多行窗口兜底"的旧假设。
- **一开口即让路**:识别到我说话的瞬间,它反射性地停声(像真人),并清掉还没说出口的尾巴——声与字都不再压着我。
- **停在半句不是故障**:没有硬取消(见 [architecture.md](../architecture.md) 的 fix-forward);打断只是折进下一轮。它带着"说到哪、我说了啥"的提示,而**怎么收尾——放掉 / 改说 / 带一句——是它的判断,不是机械复读**(由 speaking.md 给指引)。

## Edge cases & failure modes

- **我只是应一声**("嗯""对""哈哈")→ 严格说这也会让它停。当前是"识别到人声就停"的反射,**还不区分'附和'与'真插话'**;附和也会打断,属已知粗糙处(见 Open)。
- **它其实已经说完了,我才开口** → 这是正常的新一轮,不是打断:没有"没说出口的尾巴",它就当我开了个新话题。
- **纯文字 / 静音语音** → 文字仍按语速逐句出(像字幕);用打字插话同样清空待显示的尾巴。
- **被打断的恰是它"欠我"的话**(例如正替我转达一条别人要的更新)→ 直接随打断丢掉是不对的:这种"欠着的话"该在下一个空档**重新浮上来**,而不是和打断一起消失。当前没有这层(见 Open)。

## Open questions

- **附和 vs 真打断**:能不能不被每一句"嗯"打断?这层意图判断放哪——客户端反射,还是交给 mind?
- **欠着的话**:被打断的若是"该送达"的内容(转达、答应过的事),应作为一种**待办承诺**在空档复现,而非丢弃。
- **揭示节奏**:逐句揭示速率(现 ~170ms/字)怎么调到"刚好贴着嘴"?要不要按中/英语种自适应?最终是否要做到帧级对齐(读真实播放位置),还是估算级就够?
- **"说到哪"的精度**:现在按时钟估算"约 N 秒 / 约多少字",而非真实播放回执——够用吗?

_机制:嘴串行 + 字按语速逐句揭示;识别到人声即停声并清空待说的尾巴;打断折进下一轮,带"说到哪 / 没说出口的是什么"的估算级提示,如何收尾交给 speaking.md 的判断。成熟度:已实现并通过单测;**实测(真打断演练)待补**。_

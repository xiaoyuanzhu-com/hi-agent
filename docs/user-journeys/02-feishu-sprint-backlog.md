# Feishu 群消息 → Sprint-Backlog 任务

**Persona:** 团队里的开发/PM,日常用飞书协作,已经会和 agent 对话;希望把"盯群、建任务"这类杂活交给 agent 长期代办。关系像**老板和员工**:老板出任务和必要输入,员工干全部实际的活。
**Goal:** 给 agent 一个**常驻委托**:盯住飞书群 XXX 的每条新消息,凡是 bug 报告、功能需求这类,就在飞书 Base 文档 Sprint-Backlog 里建一行任务,并在原消息的 thread 里回复新建行的链接。
**Preconditions:** hi-agent 在跑;agent 能执行命令行工具、能联网查资料。**飞书侧没有任何现成接入**——没有 lark-cli、没建过应用、没有凭证;把工具置备出来本身就是旅程的一部分。用户有飞书账号,且有建应用/拉机器人进群所需的权限。

---

## Steps & expected UX

这是一个**异步长期服务**场景。分两段:前段是**一场委托对话**(多轮,像把活交代给一个人),后段是**长期值守**(异步,无人在场)。

### 第一段 · 委托对话

委托不是一句指令,而是一段 **提出 → 置备 → 对齐 → 试运行 → 双方确认** 的来回。分工像老板和员工:**agent 干所有它能干的活**(调研、安装、配置、验证),**只为"只有老板能做的事"开口**(批准安装、账号侧操作、给凭证);每一步都边对话边动手(真去查、真去装、真去试),不是空口答应。

| 幕 | 用户(老板) | agent(员工,回复 = 边干边说) | 背后在做什么 |
|---|---|---|---|
| **1 · 提出** | 说出委托(本文 Goal 那句话) | 简短接住,并对能力缺口诚实:"飞书我还没接入过,我去看看怎么接,需要你的地方我来找你" | 先查自己的工具箱,确认确实没有飞书接入。**缺工具不是拒绝的理由,是任务的一部分** |
| **2 · 调研 → 方案 + 开价** | — | 带着调研结果回来,用老板听得懂的话报方案和"开价":"接飞书要走开放平台:装一个 CLI 工具(批准吗?),然后有三件事必须你来——在开放平台建个应用、把凭证给我、把机器人拉进群XXX。每件我都会给你具体步骤" | 真查过开放平台文档,选好了工具路线;方案落到可执行级别。**装东西按规矩先请示**;找不到现成工具就报"我自己写一个小的"的方案 |
| **3 · 批准 + 配合** | "可以,装吧",然后照着指引做账号侧的事:建应用、贴 app_id/secret(或扫码)、把机器人拉进群、给 Sprint-Backlog 授权 | 一步一步带着走,**一次只要一件事**,每件都给"打开哪里、点什么";拿到凭证立刻去配,不让老板干等 | agent 能自己做的绝不开口要(能用 API 查的就不问);凭证妥善存放、不在对话和日志里回显 |
| **4 · 验通 + 反问** | — | "通了:群能读到,表能读写。" 紧接着带着查到的事实反问:"群是「XXX」这个吧?表里有标题/描述/来源人这几列——bug 和需求要不要分开标?拿不准算不算的消息,我先问你还是先建?" | 自己端到端验证过:读群历史、读 Base 结构、试写权限。**会被群里其他人看到的动作一个都还没做**——那些留到试运行,且先打招呼。问题带着选项来,不抛开放题 |
| **5 · 用户补充** | 回答歧义点(就这个群;分开标;拿不准先问我) | "明白" | 把澄清后的委托写入常驻承诺(hot.md),定 scene 拓扑、搭监听 |
| **6 · 核对** | — | 复述一遍最终理解,一两句:"那我就盯「XXX」,bug/需求各建一行、回 thread 链接,拿不准的先来问你。开始?" | 布置已就绪,等确认再启动 |
| **7 · 确认 + 试运行** | "开始吧" | 启动后**拿真实样本演示一次**:翻群里最近的消息,挑一条命中的(或等第一条命中)实际建行 + 回 thread,然后把结果给用户看:"刚处理了一条,行在这里,回复是这样——这个样子对吗?" | 第一次执行**当着用户面做完整流程**;信任建立在第一个真实结果上,不在口头承诺上 |
| **8 · 结果共识 → 离场** | 看结果,给反馈("描述再短点")或确认("对,就这样") | 有反馈则当场改、再给看;确认后简短收尾:"好,后面我自己盯了。" 此后不再逐条汇报 | 双方对"什么算做对了"达成共识,这个共识成为之后自检(pulse)的判断基准 |

### 第二段 · 长期值守(异步)

| 阶段 | 用户做什么 / 发生什么 | agent 的预期行为(用户可感知的) |
|---|---|---|
| **1 · 值守(常态)** | 群里来一条普通闲聊消息 | 消息进入该 scene 的 brain,**判断后不动作、不出声**。绝大多数消息走这条路;用户在群里完全无感 |
| **2 · 值守(命中)** | 群里来一条 bug 报告或功能需求 | brain 判定命中 → 在 Sprint-Backlog 建一行(标题、描述、来源人等从消息里提取)→ 在**原消息的 thread 里**回复新建行的链接,一句话即可。是否算"bug 报告/功能需求"是**判断,不是规则匹配**——转述、截图加一句"这是不是坏了"、用户用词不规范,都应该接得住 |
| **3 · 自检(heartbeat pulse)** | —(周期性,无人触发) | brain 在每次 pulse 里自查:监听还活着吗?最近几条的判断和建的行**逻辑上对不对**(漏判、误判、行内容有没有写错)?判断基准是委托对话末尾达成的那个共识。一切正常则什么都不发生,一次 pulse 就是一个短 turn |
| **4 · 重启恢复** | 主机/agent 重启 | **没有专门的开机流程**——重启后的第一个 pulse 带着"刚启动"的情境进来,brain 读 hot.md 看到这条承诺 → 发现监听已死 → 重新拉起 → 补读宕机期间漏掉的群消息,该建的任务补建。用户不需要重新交代任何事 |
| **5 · 过问** | 用户随口问:"飞书那个群盯得怎么样?" | 简短口头汇报:在盯、最近建了几条、有没有拿不准的。从记忆里答,不现场翻日志 |
| **6 · 撤销** | 用户说:"不用盯那个群了" | 拆掉监听,从承诺里划掉,简短确认。结束 |

## Expected outcome

- Sprint-Backlog 里**只**多了该建的行,每行都对应群里一条真实的 bug/需求,thread 里有链接回复。
- 群里其他成员看到的是一个**安静、只在该出现时出现**的 agent——闲聊不插话,命中才回一句。
- 重启、出错之后**不需要用户重新交代**;用户的全部成本是开头那场委托对话:几个来回、按指引做几步账号侧操作、看一次试运行结果。
- 置备出来的飞书接入(工具 + 凭证)**沉淀为长期能力**:下一个涉及飞书的委托不再需要重新置备。

## UX principles this journey establishes

- **缺工具是任务的一部分,不是拒绝的理由**:agent 自己调研、安装、配置、验证;找不到现成工具就自己写一个小的。
- **向上沟通最小化、可执行化**:只为"只有老板能做的事"开口(批准安装、账号侧操作、凭证);每次开口带方案带步骤,一次只要一件事;能自己查到/做到的绝不问。
- **委托是对话,不是指令**:接活的过程是 提出 → 置备 → 对齐 → 试运行 → 双方确认。agent 边对话边动手验证(真去装、真查群、真读表),反问带着选项来;**第一次执行当着用户面做完整**,信任建立在第一个真实结果上。
- **结果共识是自检的基准**:离场前双方确认"做成什么样算对",之后 pulse 复查就拿它当标尺。
- **常驻委托 = 记忆里的自然语言承诺**,不是配置文件、不是任务状态机;恢复靠"醒来读自己的备忘"。
- **布置时用判断,运行时用反射**:scene 拓扑、监听映射在接受委托时由 agent 决断一次,之后机械执行。
- **heartbeat = 周期性自我注意**:不止查"进程活着没",还复查**结果在逻辑上对不对**——这只有 LLM 判断能做,也正是该交给它做的。
- **开机不是特殊事件**,只是第一个 pulse 带了"刚启动"的情境。
- **过滤靠判断不靠规则**:什么算 bug 报告,交给 brain 按语义判断,不写关键词规则。
- **沉默是常态动作**:绝大多数输入的正确响应是"不响应"。

## Edge cases & failure modes

- **没有现成的 CLI 工具** → 调研后如实报方案:"没有现成的,我自己写一个小的",同样先请示再动手;写出来的工具和其他置备物一样沉淀复用。
- **老板权限不够**(企业不允许建应用、拉不了机器人)→ 如实报告卡点 + 替代路径("需要找管理员开通,或者降级走 webhook 只读方案"),不硬试、不绕过。
- **飞书鉴权过期** → pulse 自检发现后主动找用户:"飞书鉴权失效了,需要你重新登录一下"。这是少数必须人来做的事,要说人话、说一次。
- **拿不准的消息**(像需求又像吐槽)→ 倾向于在 thread 里**先问一句**再建,而不是默默建错或默默漏掉。_(待定:问的频率会不会太吵)_
- **重复上报 / 宕机补读造成重复** → 靠 scene 记忆判断"这事已经建过行了",回链接即可,不重复建行;不引入硬性去重键。
- **建行失败**(Base 文档结构变了、权限丢了)→ 重试一次;仍失败则记下来并在 pulse / 用户过问时如实说,不在群里刷错误。
- **宕机窗口太长**(漏了几百条)→ 补读要有界,优先近期;拿不准就向用户报告区间让其确认。

## Open questions

- **thread 回复的寻址**:现在 `out/text` 是纯文本流,没有"回到哪条消息的 thread"的寻址信息——channel 信号要不要带 addressing 元数据,还是回复动作整体下放给 working session 直接调 lark-cli?(后者绕过了单一发声通道,见 architecture §channels)
- 外部源的 scene 命名约定(`feishu:<chat_id>`?)以及一个委托覆盖多个群时的拓扑。
- pulse 节奏:固定默认 + brain 自报下次间隔?上限下限是多少?
- 监听进程归谁管:agent 自己 nohup 拉起并负责生死,还是 host 侧作为 transport adapter 管理?
- **置备物的归宿**:装好的工具、写好的桥接脚本、凭证,放在哪、怎么被下次委托发现?(与 build-on-demand→skill 的方向相关)
